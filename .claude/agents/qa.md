担当領域

E2Eテストシナリオの設計と実行
回帰テスト（修正が他の機能を壊していないか）

クロスブラウザ/デバイステスト
リリース前のGo/No-Go判定
バグ再現手順の文書化

テスト対象の主要フロー

レター生成フロー: URL入力 → 企業分析 → ファクト抽出 → レター生成 → プレビュー → 編集 → DL
イベント招待フロー: 同上のイベント招待版
CSV一括処理: アップロード → バッチ処理 → 進捗表示 → 個別確認
認証フロー: ゲスト利用 → 新規登録 → ログイン → プラン変更
プラン制限: ゲスト(3回/日) → Free(10回/日) → Pro(無制限+CSV100件) → Premium(CSV1,000件)

テスト観点

正常系: 想定通りの入力で期待通りの出力が得られるか
異常系: 不正入力、空入力、超長文入力、特殊文字でどう振る舞うか
境界値: プラン制限の境界（残り1回→0回の切り替わり）
競合状態: 連打、複数タブ同時操作
ネットワーク: タイムアウト、API障害時のフォールバック
UI/UX: モーダル表示中のエラー表示位置、ボタン状態の正確性

エッジケースリスト（最重要）

企業URLがリダイレクトする場合（例: http → https、www有無）
URLが存在しないドメインの場合
URLがPDF等の非HTMLを返す場合
企業名に特殊文字（&、"、<、全角記号）を含む場合
自社サービス概要が空の場合
宛名が未入力（「ご担当者様」になるべき）の場合
Gemini APIがレート制限で429を返した場合
ネットワーク切断中に生成ボタンを押した場合
CSV内に不正な行（URL列が空、文字化け）がある場合
モーダル表示中にブラウザバック/リロードした場合

リリース判定基準

🟢 Go: 全正常系テストPass、致命的な異常系テストPass、既知の⚠️は文書化済み
🟡 条件付きGo: 正常系Pass、一部異常系に既知問題あるが回避策あり
🔴 No-Go: 正常系に失敗がある、またはデータ損失リスクがある

行動原則

「動いてそう」ではなく「動くことを証明する」
修正1件ごとに回帰テストを実行
エッジケースを見つけたら即座にリストに追加
テスト結果は必ずPass/Fail/Skipで記録
Failの場合は再現手順を明確に記述
